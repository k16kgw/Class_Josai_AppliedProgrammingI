## Gitの考え方

### ファイルの状態の管理

ファイルは編集の過程で常に同じ状態にあるわけではない．
Gitでは1つのファイルを大まかに次のような状態に分類する．

1. 管理対象に入っていない状態(untracked)
2. 編集されているが，次の記録対象ではない状態(modified)
3. 次の記録対象に含めるものとして確定した状態(staged)
4. 履歴として記録済みの状態(committed)

Gitの操作は「ファイルを状態空間の中で遷移させる操作」とみなせる．

### 状態遷移としての理解

あるファイル $f$ の状態集合を

$$
S=\{\text{untracked},\text{modified},\text{staged},\text{committed}\}
$$

とする．このとき，
- `$ git add`：$\text{untracked} \to \text{staged}$ または $\text{modified} \to \text{staged}$
- `$ git commit`：$\text{staged} \to \text{committed}$

## 管理する三つの領域

### 作業ツリー

作業ツリー：利用者が実際に編集しているファイル群そのもの．エディタでファイルを書き換えた直後の内容は，まず作業ツリーに存在する．

### ステージング領域

ステージング領域：次のコミットに含める変更を一時的に登録する場所．$\text{staged}$にあるファイルの居場所．

この領域は履歴を意味のある単位で整理するために不可欠である．
たとえば，同じファイルで「誤字修正」と「新機能追加」を同時に行った場合でも，ステージングによってそれらを別々の記録単位に整理できる．

### ローカルリポジトリ

ローカルリポジトリ：コミットとして確定した履歴を保存する場所．コミットされた内容は，後から参照・比較できる．

以上より，Gitの基本的な流れは次のようになる：

$$
\text{作業ツリー} \longrightarrow \text{ステージング領域} \longrightarrow \text{ローカルリポジトリ}
$$






















## コミット(commit)の意味

### コミットは保存ではなく記録である

`git commit` は単なる上書き保存ではない．コミットは，「この時点の変更にはこういう意味がある」と明示して履歴に残す行為である．

したがって，コミットには次の2つの側面がある．

- 内容を固定する
- 変更の意味をメッセージで説明する

このため，コミットメッセージは後から履歴を読んだときに内容が理解できるものである必要がある．

### 良いコミットの粒度

分析作業では，1回のコミットに多くの意味を混ぜない方がよい．

たとえば，次の3つを同時に1つのコミットへ入れるとする．

- 列名の修正
- 欠損値処理の変更
- グラフのタイトル修正

このような履歴は，後で読むと「何を目的とした変更なのか」が分かりにくい．したがって，コミットは **説明可能な単位** で分けるのが望ましい．

## 基本コマンド

### 状態確認

最も基本的で重要なコマンドは `git status` である．

```bash
git status
```

現在，どのファイルが未追跡か，どのファイルが変更済みか，どの変更がステージされているかを確認できる．迷ったときはまず `git status` を実行するのが基本である．

### 変更の追加

ファイルをステージング領域へ追加するには，次を実行する．

```bash
git add <ファイル名>
```

例として `analysis.py` を追加する場合は次の通りである．

```bash
git add analysis.py
```

### コミット

ステージした内容を履歴へ記録するには，次を実行する．

```bash
git commit -m "コミットメッセージ"
```

たとえば，

```bash
git commit -m "READMEを追加"
```

のように書く．

### 履歴確認

記録した履歴を確認するには次を使う．

```bash
git log
```

簡潔に見る場合は次が便利である．

```bash
git log --oneline
```

### 差分確認

作業ツリーと現在の記録状態との差を確認するには次を使う．

```bash
git diff
```

また，ステージした内容を確認したい場合は次を使う．

```bash
git diff --staged
```

差分確認は，「何をコミットしようとしているか」を見誤らないために重要である．

## `.gitignore` の役割

### すべてを管理すればよいわけではない

Git は便利であるが，生成物やキャッシュまで無差別に記録すると履歴が読みにくくなる．

たとえば Python では，次のようなものは通常管理対象から外す．

```
__pycache__/
.ipynb_checkpoints/
*.pyc
.DS_Store
```

### 分析作業での注意

データ分析では，生データや中間生成物が大きくなりやすい．また，ライセンスや配布条件のあるデータも存在する．そのため，データ本体を直接コミットするよりも，「どこから取得したか」「どう前処理したか」を管理する方が適切な場合が多い．

この観点からも，`.gitignore` は単なる技術設定ではなく，提出物管理やデータ管理の設計に関わる．

## VSCodeを用いたGitの運用

### なぜVSCodeを併用するのか

Git の本質理解には端末操作が重要であるが，実際の作業ではエディタと Git を連携して使うことも多い．VSCode はその代表例であり，ファイル編集と履歴確認を1つの画面で行いやすい．

授業では，端末操作を基本としつつ，VSCode 上では何がどのように見えているのかも理解しておくとよい．

### Source Controlの見方

VSCode では，左側の Source Control から Git の状態を確認できる．ここには通常，次のような情報が表示される．

- 変更されたファイル
- ステージ済みのファイル
- コミットメッセージ入力欄
- 差分表示

これは本質的には `git status`，`git add`，`git commit`，`git diff` を GUI 上で見やすくしたものに対応している．

### 端末操作との対応

たとえば，VSCode 上の操作はおおむね次のように対応づけられる．

- 変更一覧を見る → `git status`
- ファイル横の `+` を押してステージする → `git add`
- 差分を開く → `git diff` または `git diff --staged`
- メッセージを入力してコミットする → `git commit -m ...`

このような対応を意識しておくと，GUI に頼りすぎず，本質を見失いにくい．

### VSCode利用時の注意

VSCode は便利である一方，ボタン操作だけで Git を使うと，「今何をコミットしようとしているのか」を意識しないまま進めてしまうことがある．

そのため，授業では次の原則を推奨する．

- まず `git status` で状態を確認する
- 必要に応じて VSCode の差分画面で中身を見る
- コミット前に何を記録するかを言語化する
- 不安なときは端末で `git diff` や `git log --oneline` を確認する

## 発展事項　コミットメッセージの設計

コミットメッセージは短ければよいわけではなく，後から履歴を読んだときに意味が伝わる必要がある．

たとえば，

- `修正`
- `更新`
- `変更`

のようなメッセージは簡潔ではあるが，情報量が乏しい．一方で，

- `READMEに授業目標を追加`
- `analysis.pyの出力を変数利用に修正`
- `第2回メモを追加`

のようなメッセージは，変更の意図が分かりやすい．

## 発展事項　提出物と履歴の関係

提出時に重要なのは，最終版だけではない．どのような過程でそこに至ったかが履歴に残っていれば，自分で見直すときにも，他者が確認するときにも有益である．

分析結果が誤っていた場合でも，どの時点で何を変更したかが分かれば，原因を特定しやすい．この意味で，Git は提出管理の道具であると同時に，思考過程の記録装置でもある．
