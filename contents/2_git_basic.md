## Gitの考え方

### ファイルの状態の管理

ファイルは編集の過程で常に同じ状態にあるわけではない．
Gitでは1つのファイルを大まかに次の状態に分類する．

1. 管理対象に入っていない状態 (untracked)
2. 編集されているが，次の記録対象ではない状態 (modified)
3. 次の記録対象に含めるものとして確定した状態 (staged)
4. 履歴として記録済みの状態 (committed)

Gitの操作は「ファイルを状態空間の中で遷移させる操作」とみなせる．

### 状態遷移としての理解

あるファイル $f$ の状態集合を

$$
S=\{\text{untracked},\text{modified},\text{staged},\text{committed}\}
$$

とする．このとき，
- `$ git add`：$\text{untracked} \to \text{staged}$ または $\text{modified} \to \text{staged}$
- `$ git commit`：$\text{staged} \to \text{committed}$

## 管理する三つの領域

### 作業ツリー

作業ツリー：利用者が実際に編集しているファイル群そのもの．エディタでファイルを書き換えた直後の内容は，まず作業ツリーに存在する．

### ステージング領域

ステージング領域：次のコミットに含める変更を一時的に登録する場所．$\text{staged}$にあるファイルの居場所．

この領域は履歴を意味のある単位で整理するために不可欠である．
たとえば，同じファイルで「誤字修正」と「新機能追加」を同時に行った場合でも，ステージングによってそれらを別々の記録単位に整理できる．

### ローカルリポジトリ

ローカルリポジトリ：コミットとして確定した履歴を保存する場所．コミットされた内容は，後から参照・比較できる．

以上より，Gitの基本的な流れは次のようになる：

$$
\text{作業ツリー} \longrightarrow \text{ステージング領域} \longrightarrow \text{ローカルリポジトリ}
$$

```{tip} 注意
新しく作成したファイルは追跡されていないため，作業ツリー・ステージング領域・ローカルリポジトリのいずれにも属していない（untracked）．
```

## 初期設定

1. 本講義で使用するフォルダ `/User/<ユーザ名>/applied_programming_i/` 内に，本日使用するフォルダ `2` を作成する
   1. フォルダ移動（`$ cd /User/<ユーザ名>/applied_programming_i/`）
   2. フォルダ`2`を作成（`$ mkdir 2`）
   3. フォルダ`2`に移動（`$ cd 2`）
2. Gitの初期化を行う（`$ git init`）
3. 最初のコミットを行い初期状態を記録する
   1. 全てのファイルをステージング領域に上げる（`$ git add .`）
   2. コミットする（`$ git commit -m "初めてのコミット"`）

## コミット (commit)

- コミット（`$ git commit`）：この時点の変更にはこういう意味があると明示して履歴に残す行為

コミットには次の2つの側面がある．

- 内容を固定する
- 変更の意味をメッセージで説明する

コミットメッセージは後から履歴を読んだときに内容が理解できるものである必要がある．

### 良いコミットの粒度

作業では1回のコミットに多くの意味を混ぜない方がよい．

たとえば，次の3つを同時に1つのコミットへ入れるとする．

- データの列名の修正
- データ内の欠損値処理の方法の変更
- グラフのタイトル修正

このような履歴は，後で読むと「何を目的とした変更なのか」が分かりにくい．
コミットを**説明可能な単位**で分けることによって後からプログラムの変更がどのような手順で行われたのかを理解しやすくなる．

## 基本コマンド

### 状態確認（`git status`）

- 現在，どのファイルが未追跡か，どのファイルが変更済みか，どの変更がステージされているかを確認する．
    ```bash
    $ git status
    ```

### 変更の追加（`git add`）

- 作業ツリーにあるファイルをステージング領域へ追加する．
    ```bash
    $ git add <ファイル名>
    ```

- 例：`analysis.py` を追加する場合
    ```bash
    $ git add analysis.py
    ```

### コミット（`git commit`）

- ステージング領域のファイルをローカルリポジトリへ記録する．
    ```bash
    $ git commit -m "コミットメッセージ"
    ```
    ※ `-m`は後ろにメッセージをつけるオプション

- 例：
    ```bash
    $ git commit -m "READMEを追加"
    ```

### 履歴確認（`git log`）

- 記録した履歴を確認する．
    ```bash
    $ git log
    ```
    出力例
    ```bash
    commit a393e74a87e0392f2cf123a5a9c3a3f98d3bef00 (HEAD -> master)
    Author: komachan <kkagawa@josai.ac.jp>
    Date:   Tue Apr 21 00:22:15 2026 +0900

    make README
    ```

- 簡潔に見る場合
    ```bash
    $ git log --oneline
    ```

### 差分確認（`git diff`）

- 作業ツリーと最新のコミット(HEAD)との差分を確認する．
    ```bash
    $ git diff
    ```

- ステージング領域と最新のコミット(HEAD)との差分を確認する．
    ```bash
    $ git diff --staged
    ```

## gitignoreファイル

Git は履歴管理としては便利だが，生成物やキャッシュまで無差別に記録すると履歴が読みにくくなる．

次のファイルは通常，管理対象から外し，常にuntrackedとして扱う．
- 自動生成ファイル：プロジェクトを動作するときに自動で生成されるので管理しない
- 環境依存ファイル：プロジェクト自体はOSがmacであることは関係ないため管理しない

自動生成ファイルの例：
```
__pycache__/
.ipynb_checkpoints/
*.pyc
```

環境依存ファイルの例：
```
.DS_Store
```

管理対象から外すファイルを指定するファイルが `.gitignore` であり，例えば次のように記載する．

```.gitignore
__pycache__/
.ipynb_checkpoints/
*.pyc
.DS_Store
```

### 分析作業での注意

データ分析では生データや中間生成物の数と容量が大きくなりやすい．
また，ライセンスや配布条件のあるデータも存在する．
そのため，データ本体を直接コミットするよりも，「どこから取得したか」「どう前処理したか」を管理する方が適切な場合が多い（再現性を確保するための最低限の情報を管理する）．

この観点からも `.gitignore` は提出物管理やデータ管理の設計に関わる．

## VSCodeでのGitの運用

ファイル管理の作業ではエディタと Git を連携して使うことも多い．
VSCode ではファイル編集と履歴確認を1つの画面で行いやすい．

VSCodeの左のタブから次のマークを選択するとGitの機能を確認できる．

![git](./figs/2/vscode0.png)

### ソース管理の見方

VSCode では，左側のソース管理から Git の状態を確認できる．ここには通常，次のような情報が表示される．

- コミットメッセージ入力欄
- ステージ済みのファイル
- 変更されたファイル
- 差分表示（表示されているファイルを選択すると差分表示でファイルを確認できる）

これは `git status`，`git add`，`git commit`，`git diff` を GUI 上で見やすくしたものに対応している．

![git](./figs/2/vscode1.png)

### 端末操作との対応

たとえば VSCode 上の操作はおおむね次のように対応づけられる．

- 変更一覧を見る → `git status`
- ファイル横の `+` を押してステージする → `git add`
- 差分を開く → `git diff` または `git diff --staged`
- メッセージを入力してコミットする → `git commit -m ...`

## コミットメッセージの設計

コミットメッセージは短ければよいわけではなく，後から履歴を読んだときに意味が伝わる必要がある．

たとえば，

- `修正`
- `更新`
- `変更`

のようなメッセージは簡潔ではあるが，情報量が乏しい．一方で，

- `READMEに授業目標を追加`
- `analysis.pyの出力を変数利用に修正`
- `第2回メモを追加`

のようなメッセージは，変更の意図が分かりやすい．

## まとめ

- Git ではファイルは状態を持ち，操作によって遷移する
- 作業ツリー，ステージング領域，ローカルリポジトリを区別して理解できる
- 基本操作：`git status`，`git add`，`git commit`，`git log`，`git diff`
- コミットは小さく，意味の明確な単位で行う方がよい
- VSCode の ソース管理 は便利だが，端末操作との対応を理解して使うことが重要である
