## ブランチ (branch)

### 直線的な作業の限界

第2回までの理解では，履歴は

$$
C_0 \to C_1 \to C_2 \to \cdots \to C_n
$$

のような直線的列として表せる．ここで $C_i$ はコミットを表す．

この見方は基本操作の理解には十分であるが，実際には次のような状況に対応しづらい．

- 現在の安定版を保ったまま新機能を試したい
- 可視化の案を2通り比較したい
- 前処理方法を複数並行して検討したい
- 複数人が別々の作業を進めた後で統合したい

このようなとき，単一列しか持たない運用では，途中の試行を本流に混ぜてしまいやすい．その結果，履歴が読みにくくなり，再現性も低下する．

### 履歴を分岐させるという考え方

そこで Git では，履歴を複数の系列に分けて管理できる．これがブランチである．

ブランチを使うことで，たとえば安定版の系列を保ったまま，別の系列で新しい分析手法を試せる．うまくいった場合だけ本流に取り込み，うまくいかなければ試行用の系列を破棄できる．

これは数理的には，単線の列ではなく，有向グラフ上の複数の経路を管理することに対応する．

## Git履歴のグラフ構造

### コミットグラフ

Git の履歴は，コミットを頂点とする有向非巡回グラフとして理解できる．

コミット集合を $V$，親子関係の集合を $E$とすると，履歴は

$$
G=(V,E)
$$

と表せる．各コミットは通常1つ以上の親を持ち，辺は「新しいコミットから親コミットへ向かう」参照関係を表す．

通常の連続作業では，各コミットは1つの親を持つので，見た目は鎖のようになる．しかし分岐と統合を許すと，グラフは木状あるいはそれに近い構造を持つ．

### ブランチとは何か

初学者は，ブランチを「コミットの束そのもの」と考えがちである．しかし厳密には，ブランチとは **あるコミットを指す可変参照** と理解するのがよい．

たとえば，`main` というブランチ名があるとき，それは現在の本流の先頭コミットを指している．新しいコミットが `main` 上で作られると，`main` という参照がその新しいコミットへ移動する．

この意味で，ブランチは履歴を複製するものではなく，履歴グラフ上の位置を指し示すラベルである．

### HEADとは何か

Git では，現在自分がどのブランチやコミットを基準に作業しているかを示す特別な参照として `HEAD` がある．

通常は `HEAD` は現在のブランチ名を経由して，その先頭コミットを指す．したがって，ブランチを切り替えることは，`HEAD` が参照する先を変えることに対応する．

概念的には，

$$
\text{HEAD} \to \text{branch name} \to \text{commit}
$$

という参照の鎖を考えると理解しやすい．

## ブランチ操作の理論的理解

### 局所的試行の分離

分析作業の状態を $S$ とし，ある変換を $T$ とする．単純な逐次作業では

$$
S_{t+1}=T_t(S_t)
$$

のように状態更新が進む．しかし現実には，同じ $S_t$ から異なる候補変換 $T_t^{(1)}$ と $T_t^{(2)}$ を試したいことがある．このとき，

$$
S_{t+1}^{(1)}=T_t^{(1)}(S_t), \qquad
S_{t+1}^{(2)}=T_t^{(2)}(S_t)
$$

という2つの後続状態が生まれる．

ブランチは，このような「同じ出発点から複数の候補系列を展開する」ための機構と解釈できる．情報数理の観点では，探索木や状態空間探索にも似た発想である．

### 参照の移動としてのコミット

現在のブランチを $b$，その指すコミットを $c$ とする．このブランチ上で新たなコミット $c'$ を作ると，ブランチ参照は

$$
b : c \mapsto c'
$$

のように更新される．

ここで重要なのは，過去のコミット $c$ が消えるのではなく，新しいコミット $c'$ が親として $c$ を参照し，ブランチの先端だけが前進するという点である．この設計により，過去の履歴は保持される．

## マージの意味

### 分岐した履歴の再統合

2つのブランチが共通の祖先から分岐していたとする．その後，片方の変更をもう片方へ取り込みたい場合に使うのがマージである．

概念的には，ある共通祖先 $C_0$ から

$$
C_0 \to C_1 \to C_2
$$

と

$$
C_0 \to D_1 \to D_2
$$

という2系列が生じているとき，これらを統合した新たなコミット $M$ を作る操作がマージである．

このとき，$M$ は通常2つの親を持つ．したがって，マージコミットはグラフ構造の上で特別な役割を持つ．

### fast-forward とマージコミット

もし片方のブランチが，もう片方の単なる先行版であるなら，明示的なマージコミットを作らずに参照を前に進めるだけで統合できることがある．これが fast-forward である．

一方，双方で独立にコミットが進んでいた場合には，通常は新しいマージコミットが必要になる．

この違いは，グラフ理論的には「一方が他方の祖先かどうか」に対応する．祖先関係が成り立つなら参照移動だけで済み，そうでなければ2親を持つ統合点が必要になる．

## コンフリクトの本質

### コンフリクトはなぜ起こるか

マージ時にコンフリクトが起きるのは，同じ箇所に対して両系列が両立しない変更を加えているからである．

より形式的には，共通祖先からの差分を2系列で比較したとき，同一の対象に対して異なる更新が定義され，Git が自動的に一意な統合結果を決められない場合にコンフリクトが生じる．

たとえば，共通祖先で同じ1行だった部分を，片方では文言Aに，もう片方では文言Bに変えていた場合，どちらを採用するべきかは文脈依存である．そのため，最終判断を人間に委ねる．

### コンフリクト解消の考え方

コンフリクト解消は，単に印を消す作業ではない．複数の変更候補のうち，どの内容が最終的に望ましいかを意味的に判断する過程である．

この点はデータ分析において特に重要である．片方では欠損値処理の基準を変更し，もう片方では列名の整理を行っていた場合，両方を適切に残すべきこともあるし，一方のみを採用すべきこともある．

したがって，コンフリクト解消は技術操作であると同時に，分析方針を明示化する判断でもある．

## データ分析におけるブランチ運用

### 安定版と試行版の分離

データ分析では，常に「今ある再現可能な版」を保持しておくことが重要である．一方で，探索的な試行も不可欠である．

そこで，たとえば次のような役割分担が考えられる．

- `main`：再現可能で報告可能な安定版
- `feature/cleaning-rule`：前処理規則の試行
- `feature/new-visualization`：新しい可視化案
- `experiment/model-comparison`：モデル比較の実験系列

このように役割を分けると，分析の信頼性と柔軟な試行錯誤を両立しやすい．

### 履歴の可読性

ブランチを用いずにすべての試行を本流へ直接積み重ねると，履歴は雑然としやすい．その結果，「最終結果に関係ある変更」と「途中で捨てた試行」が混ざってしまう．

ブランチ運用をすると，試行のまとまりが明確になる．これは後から履歴を読む際に，「何のための変更系列だったのか」を理解しやすくする．

### 再現性との関係

再現性の観点では，報告や提出に対応するコミット列が明確であることが重要である．ブランチ運用により，探索的な変更を分離すれば，最終報告に対応する系列を安定して保持できる．

これは，「探索」と「確定」を混同しないという研究的作法にも通じる．

## 基本コマンド

### ブランチの確認

現在存在するブランチを確認するには，次を実行する．

```bash
git branch
```

### ブランチの作成

新しいブランチを作成するには，次を実行する．

```bash
git branch feature-notes
```

### ブランチの切り替え

既存ブランチへ切り替えるには，次を実行する．

```bash
git switch feature-notes
```

古い書き方として `git checkout` もあるが，本授業では意味が分かりやすい `git switch` を主に使う．

### ブランチ作成と切り替えを同時に行う

新しいブランチを作成し，そのまま切り替えるには，次を実行する．

```bash
git switch -c feature-notes
```

### マージ

現在いるブランチへ，別ブランチの変更を取り込むには，たとえば `main` 上で次を実行する．

```bash
git merge feature-notes
```

### グラフ付き履歴表示

履歴の分岐を確認するには，次が有用である．

```bash
git log --oneline --graph --all
```

これは第3回で特に重要なコマンドである．ブランチとマージを視覚的に理解しやすくなる．

## 発展事項　ブランチ運用の設計

ブランチを使えばよいというだけでは，よい運用にはならない．重要なのは，どのような粒度で分岐し，どの時点で統合するかである．

粗すぎる運用では，1つのブランチに多くの目的が混ざってしまう．逆に細かすぎる運用では，管理コストが増え，何のためのブランチなのか分かりにくくなる．

授業レベルでは，次のような原則が扱いやすい．

- 1つのブランチには1つの主目的を持たせる
- ある程度まとまったら早めにマージする
- マージ前に `git log --graph --all` や差分確認を行う
- 安定版に対応する `main` は常に説明可能な状態に保つ

## 発展事項　マージと知識統合

情報数理的な観点では，マージは単なるファイル統合ではなく，分岐した知識や試行の統合ともみなせる．

たとえば，一方の系列では前処理条件を調整し，もう一方では可視化方法を改善していたとする．それらを統合する操作は，互いに独立な改善を1つの成果物へまとめることである．

一方で，同じ対象に対して異なる仮説的修正を加えていた場合には，統合には判断が必要になる．この意味で，マージは形式的な操作であると同時に，研究や開発における意思決定でもある．

重要な点は次の通りである．

- Git の履歴は単なる直線ではなく，有向非巡回グラフとして理解できる
- ブランチは履歴の複製ではなく，先頭コミットを指す可変参照である
- `HEAD` は現在の作業位置を示す重要な参照である
- マージは分岐した履歴を再統合する操作であり，必要に応じて2親を持つコミットが作られる
- コンフリクトは，同じ対象への両立しない変更が原因で生じる
- データ分析では，安定版と試行版を分けることで，再現可能性と探索性を両立できる
