## データ取得で確認すべき事項

分析に用いるデータを取得するときには，少なくとも次の点を確認すること．

- 出典
- ライセンス
- 更新頻度
- 取得時点
- データ形式
- 変数の意味と単位

### 出典

**どの機関，団体，研究プロジェクト，企業などが提供しているデータであるか**を確認する．

出典が不明なデータは正確性や更新性を評価しにくい．  
行政機関の統計，大学研究室の公開データ，民間企業の提供データとでは，作成方法や品質管理の方針が異なる可能性がある．

### ライセンス

ライセンス：そのデータをどこまで利用してよいかを定める条件

特に次の点を確認する．

- 商用利用が可能か
- 改変が許可されているか
- 再配布が可能か
- 出典表示が必要か
- 利用してはいけない用途があるか

### 更新頻度と取得時点

データは常に固定されているとは限らない．  
特に，気象，交通，経済，人口，感染症などのデータは継続的に更新される．

同じプログラムを使ったデータ分析結果であったとしても，データを取得した日時が異なると異なる分析結果が出る可能性がある．
<!-- そのため，データ分析の際には，いつ取得したデータであるかを記録しておくことは重要である． -->

<!-- このとき，分析結果を

## 機械可読性とデータ形式

### 機械可読性

人間が読みやすい形式と，プログラムが処理しやすい形式は必ずしも一致しない．

**機械可読性**：プログラムによって処理しやすい性質

例）PDF：ファイルに掲載されている表は人間は理解しやすいが，プログラムで集計するには追加の変換が必要になることが多い．

例）CSV・JSON：構造が比較的明確であり，プログラムで扱いやすい．

### 代表的な形式

- CSV：行と列からなる表形式データ．単純で扱いやすい．
- Excel：表形式データだが，複数シート，装飾，結合セルなどを含むことがある場合には人間には読みやすいが機械には読みにくい．
- JSON：キーと値の組を基本とし，リストや入れ子構造を表現できる．
- HTML：ウェブページ表示用の構造をもつ．分析には抽出処理が必要になることがある．
- PDF：文書表示用の形式．人間には読みやすいが，機械処理には向かないことが多い．

### CSVと表データ

CSV：表形式データを扱う基本的な形式の1つ．多くの場合，各行が観測単位，各列が変数を表す．

例）市区町村別人口データ

- 行：各市区町村
- 列：都道府県名，市区町村名，人口，面積など

データ表は，行の集合 $I$ と列の集合 $J$ に対して値を返す写像
$$
X : I \times J \to V
$$
とみなせる．
ここに $V$ は値の集合である．
<!-- この見方をすると，表データは単なる文字の集まりではなく，観測と変数からなる構造体として理解できる． -->
<!-- 
### tidy data の考え方

分析しやすい表データには，次のような原則がある．

- 1つの列は1つの変数に対応する
- 1つの行は1つの観測に対応する
- 1つの表は1種類の観測単位に対応する

### JSONと階層構造

JSON：キーと値（key & value）の組を基本とする形式．
値には文字列，数値，真偽値，リスト，別のオブジェクトなどを含めることができる．
階層的な情報を表す場合に有効である．

例）ある地域の気象情報

```json
{
  "location": "Tokyo",
  "observations": [
    {"date": "2025-04-01", "temp": 18.2, "rain": 0.0},
    {"date": "2025-04-02", "temp": 16.5, "rain": 12.0}
  ]
}
```

- `location`：地点名
- `observations`：観測値のリスト

### 生データと加工データ

分析の過程では，取得直後のデータと加工後のデータを区別することが重要である．
生データに変更を直接上書きすると，どこまでが元データで，どこからが自分の加工結果なのか分からなくなる．


- `data/processed/`：前処理や集計を行った後のデータ（加工データ）
- `src/`：取得や加工に用いるプログラム（ソースファイル）
- `README.md`：出典，取得日，ライセンス，作業手順などの記録

## Git管理とデータ取得

第2回・第3回で学んだ Git は，データ取得と整理にも役立つが，データ分析では何でも Git で管理すればよいわけではない．

### Gitで管理しやすいもの

- 取得用スクリプト
- 前処理スクリプト
- README や取得手順書
- メタデータ
- 小規模なサンプルデータ

### Gitで管理しにくいもの

- サイズの大きいデータ
- 頻繁に更新される取得結果
- 個人情報や機密情報を含むデータ
- ライセンス上，再配布できないデータ

データそのものをすべて Git で管理することではなく，**どのようにデータを取得し，どのように加工したか**を記録することが重要である．

- 行数
- 列名
- 先頭の数行
- 日付の形式

````{note} 課題2：CSVの概要を確認する
`src/inspect_covid_data.py` を次の内容で作成し，CSVの概要を確認せよ．

```python
import csv

input_path = "data/raw/newly_confirmed_cases_daily.csv"

with open(input_path, encoding="utf-8-sig") as f: # ファイルを開く
    reader = csv.DictReader(f) # CSVの各行を辞書(dict)として読み込む
    rows = list(reader) # 全行をリスト化する
    fieldnames = reader.fieldnames # CSVのヘッダー行（列名）を取得

print("行数:", <行数>)
print("列数:", <列数>)
print("列名:")
print(<列名>)

print("先頭5行:")

    print(row)
```
````

†：BOM (Byte Order Mark) とはどのような文字コードで保存されたテキストかを明示する特殊バイト

## まとめ

- オープンデータとは，単に閲覧できるデータではなく，再利用条件が明示されたデータである．
- データ取得では，出典，ライセンス，更新頻度，取得時点，形式を確認する必要がある．
- CSV は表形式，JSON は階層構造を表す代表的な形式である．
- 生データと加工データを分けることで，元データと自分の処理結果を区別できる．
- 実際のオープンデータを取得したら，行数，列名，先頭行，一部の値を確認して，データの形を把握する．
- データ本体だけでなく，README やメタデータを残すことで，後から分析過程を説明しやすくなる．
- Git では，データそのものだけでなく，取得手順と整理過程を管理することが重要である．

次回は API の使い方を扱う．

- ウェブ上のデータ提供サービスから，プログラムを用いてデータを取得する方法を学ぶ．

### 小さな加工データを作成する

取得した生データには多くの列が含まれる．
ここでは，後の授業で扱いやすいように，一部の列だけを取り出した小さなCSVを作成する．

````{note} 発展演習3：加工済みデータを作成する
`src/extract_covid_columns.py` を次の内容で作成し，`data/processed/covid_selected.csv` を作成せよ．  
なお，`〇〇`には自身の好きな都道府県を選び入力せよ．

```python
import csv

input_path = "data/raw/newly_confirmed_cases_daily.csv"
output_path = "data/processed/covid_selected.csv"
columns = ["Date", "ALL", "Tokyo", "Saitama", "〇〇"]

with open(input_path, encoding="utf-8-sig") as f:
    reader = csv.DictReader(f)
    rows = list(reader)

with open(output_path, "w", encoding="utf-8", newline="") as f:
    writer = csv.DictWriter(f, fieldnames=columns)
    writer.writeheader()

    for row in rows:
        writer.writerow({column: row[column] for column in columns})

print("saved:", output_path)
```
````
