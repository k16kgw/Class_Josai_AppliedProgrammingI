## オープンデータ

**オープンデータ**：一般に，誰もが利用，複製，再配布，加工などを行えるように，利用条件が明示された形で公開されているデータ．
閲覧可能性だけでなく，再利用可能性が明示されている．

※ ある表がウェブページに掲載されていても利用条件が書かれていない場合には，そのデータを自由に分析したり，再配布したりしてよいとは限らない．
データ分析をする上では**利用条件を確認した上で使う**態度が必要である．

### オープンデータの意義

オープンデータは，行政の透明性向上，科学的検証，市民参加，教育，企業活動など，さまざまな場面で利用される．

たとえば行政統計が機械可読な形で公開されていれば，地域比較，政策評価，都市計画，市場分析などに利用できる．
気象データや交通データであれば，時系列解析，予測，混雑分析，リスク評価などに活用できる．

この意味で，オープンデータは単なる分析素材ではなく，社会の意思決定や知識生成を支える基盤である．

---

## データ取得で確認すべき事項

分析に用いるデータを取得するときには，少なくとも次の点を確認すること．

- 出典
- ライセンス
- 更新頻度
- 取得時点
- データ形式
- 変数の意味と単位

### 出典

単なる URL だけではなく，**どの機関，団体，研究プロジェクト，企業などが提供しているデータであるか**を確認すること．

出典が不明なデータは正確性や更新性を評価しにくい．
行政機関の統計，大学研究室の公開データ，民間企業の提供データでは，作成方法や品質管理の方針が異なる可能性があるので利用する際に注意すること．

### ライセンス

ライセンス：そのデータをどこまで利用してよいかを定める条件

特に次の点を確認する．

- 商用利用が可能か
- 改変が許可されているか
- 再配布が可能か
- 出典表示が必要か
- 利用してはいけない用途があるか

### 更新頻度と取得時点

データは常に固定されているとは限らない．
気象，交通，経済，人口，感染症などのデータは継続的に更新される．

同じプログラムを使ったデータ分析結果であったとしても，データを取得した日時が異なると，異なる分析結果が出る可能性がある．
そのため，データ分析の際には，いつ取得したデータであるかを記録しておくことは重要である．

<!-- このとき，分析結果を

### データ形式

同じ内容のデータでも，CSV，JSON，Excel，HTML，PDF など，形式が異なれば処理方法は変わる．
形式は単なる拡張子ではなく，**データの構造**を表している．

分析者は，データに何が書かれているかだけでなく，どのような構造で保存されているかを理解する必要がある．

---

## 機械可読性とデータ形式

### 機械可読性

人間が読みやすい形式と，プログラムが処理しやすい形式は必ずしも一致しない．

**機械可読性**：プログラムによって処理しやすい性質

例）PDF：ファイルに掲載されている表は人間は理解しやすいが，プログラムで集計するには追加の変換が必要になることが多い．

例）CSV や JSON：構造が比較的明確であり，プログラムから扱いやすい．

### 代表的な形式

- CSV：行と列からなる表形式データ．単純で扱いやすい．
- Excel：表形式データだが，複数シート，装飾，結合セルなどを含むことがある場合には人間には読みやすいが機械には読みにくい．
- JSON：キーと値の組を基本とし，リストや入れ子構造を表現できる．
- HTML：ウェブページ表示用の構造をもつ．分析には抽出処理が必要になることがある．
- PDF：文書表示用の形式．人間には読みやすいが，機械処理には向かないことが多い．

入手可能な形式が複数ある場合は，分析しやすく，機械可読性の高い形式を選ぶことが望ましい．

## CSVと表データ

### 行と列

CSV は，表形式データを扱う基本的な形式の1つである．
多くの場合，各行が観測単位，各列が変数を表す．

たとえば，市区町村別人口データであれば，

- 行：各市区町村
- 列：都道府県名，市区町村名，人口，面積など

のように対応する．

このとき，データ表は，行の集合 $I$ と列の集合 $J$ に対して値を返す写像
$$
X : I \times J \to V
$$
とみなすことができる．
ここに $V$ は値の集合である．
<!-- この見方をすると，表データは単なる文字の集まりではなく，観測と変数からなる構造体として理解できる． -->















### tidy data の考え方

分析しやすい表データには，次のような原則がある．

- 1つの列は1つの変数に対応する
- 1つの行は1つの観測に対応する
- 1つの表は1種類の観測単位に対応する

この考え方は tidy data と呼ばれる．
取得した直後のデータは，必ずしもこの形になっているとは限らない．
したがって，分析の前には，表の構造を確認し，必要に応じて整理する必要がある．

## JSONと階層構造

JSON は，キーと値の組を基本とする形式である．
値には文字列，数値，真偽値，リスト，別のオブジェクトなどを含めることができる．

たとえば，ある地域の気象情報を次のように表せる．

```json
{
  "location": "Tokyo",
  "observations": [
    {"date": "2025-04-01", "temp": 18.2, "rain": 0.0},
    {"date": "2025-04-02", "temp": 16.5, "rain": 12.0}
  ]
}
```

このデータでは，`location` は地点名を表し，`observations` は観測値のリストを表す．
JSON は，単純な表よりも階層的な情報を表しやすい．

情報数理の観点では，JSON は木構造として理解できる．
オブジェクトを節点，キーを枝，値を葉または部分木とみなせば，JSON の処理は「木の中から必要な情報を取り出す操作」と考えられる．

## データの保存と整理

### 生データと加工データ

分析の過程では，取得直後のデータと，加工後のデータを区別することが重要である．
たとえば，次のようなディレクトリ構成が考えられる．

```text
project/
├── data/
│   ├── raw/
│   └── processed/
├── src/
└── README.md
```

- `data/raw/`：取得直後のデータ
- `data/processed/`：前処理や集計を行った後のデータ
- `src/`：取得や加工に用いるプログラム
- `README.md`：出典，取得日，ライセンス，作業手順などの記録

生データを直接上書きすると，どこまでが元データで，どこからが自分の加工結果なのか分からなくなる．
そのため，元データは `raw` に保存し，加工結果は `processed` に保存するのが望ましい．

### メタデータ

データ本体だけでは，分析に必要な情報が十分に分からないことがある．
たとえば，変数名だけを見ても単位や定義が分からない場合がある．

このような，データを説明するためのデータを**メタデータ**という．

メタデータとしては，次のような項目を記録する．

- ファイル名
- データ名
- 出典
- 取得日
- 最終更新日
- ライセンス
- 変数の意味
- 単位
- 欠損値の扱い
- 注意点

## Git管理とデータ取得

第2回・第3回で学んだ Git は，データ取得と整理にも役立つ．
ただし，データ分析では何でも Git に入れればよいわけではない．

### Gitで管理しやすいもの

- 取得用スクリプト
- 前処理スクリプト
- README や取得手順書
- メタデータ
- 小規模なサンプルデータ

### Gitで管理しにくいもの

- サイズの大きいデータ
- 頻繁に更新される取得結果
- 個人情報や機密情報を含むデータ
- ライセンス上，再配布できないデータ

重要なのは，「データそのもの」をすべて Git に入れることではなく，「どのようにデータを取得し，どのように加工したか」を記録することである．

データ取得の操作は，次のように表すことができる．
$$
D = G(u, t, \theta)
$$
ここで，

- $u$：取得元
- $t$：取得時点
- $\theta$：取得条件
- $D$：取得されたデータ

である．
取得元，取得時点，取得条件を記録しておけば，後から同様の取得を再実行しやすくなる．

## データの加工

取得したデータをそのまま使うだけでなく，分析しやすい形に加工することがある．
ここでは，人口と面積から人口密度を計算する．

````{note} 演習4
`population.csv` をもとに，人口密度を追加した `data/processed/population_processed.csv` を作成する．
`src/process_population.py` を次の内容で作成せよ．

```python
import csv

input_path = "data/raw/population.csv"
output_path = "data/processed/population_processed.csv"

rows_out = []

with open(input_path, encoding="utf-8") as f:
    reader = csv.DictReader(f)
    for row in reader:
        population = int(row["population"])
        area = float(row["area"])
        density = population / area
        rows_out.append({
            "prefecture": row["prefecture"],
            "city": row["city"],
            "population": population,
            "area": area,
            "density": density
        })

with open(output_path, "w", encoding="utf-8", newline="") as f:
    writer = csv.DictWriter(
        f,
        fieldnames=["prefecture", "city", "population", "area", "density"]
    )
    writer.writeheader()
    writer.writerows(rows_out)

print("processed data saved.")
```

実行後，`data/processed/population_processed.csv` が作成されていることを確認せよ．
````

## データ取得の倫理と注意点

### 利用条件を確認する

公開されているからといって，どのように使ってもよいとは限らない．
特に，個人情報やセンシティブな情報を含むデータには注意が必要である．

授業では主に公開データを扱うが，公開データであっても，利用条件や再配布条件を確認する必要がある．

### データ品質を確認する

オープンデータは有用である一方で，欠損，表記揺れ，更新遅れ，定義変更などを含むことがある．

分析者は，「公開されているから正しい」と無批判に受け入れるのではなく，データがどのように作られ，どのような限界をもつかを確認しなければならない．

## まとめ

- オープンデータとは，単に閲覧できるデータではなく，再利用条件が明示されたデータである．
- データ取得では，出典，ライセンス，更新頻度，取得時点，形式を確認する必要がある．
- CSV は表形式，JSON は階層構造を表す代表的な形式である．
- 生データと加工データを分けることで，元データと自分の処理結果を区別できる．
- データ本体だけでなく，README やメタデータを残すことで，後から分析過程を説明しやすくなる．
- Git では，データそのものだけでなく，取得手順と整理過程を管理することが重要である．

## 自主学習用の発展問題

````{note} 発展課題1：身近なオープンデータを調べる

自分が関心をもつオープンデータを1つ選び，次の表を埋めよ．

| 項目 | 内容 |
| --- | --- |
| データ名 | ここに記入 |
| 出典機関 | ここに記入 |
| URL | ここに記入 |
| ライセンス | ここに記入 |
| データ形式 | ここに記入 |
| 更新頻度 | ここに記入 |
| 分析してみたい内容 | ここに記入 |

その上で，そのデータが分析に使いやすいかどうかを，形式，変数説明，更新頻度，ライセンスの観点から説明せよ．
````

```{dropdown} 解答例

| 項目 | 内容 |
| --- | --- |
| データ名 | 市区町村別人口データ |
| 出典機関 | 国または自治体の統計提供機関 |
| URL | データが掲載されているページの URL |
| ライセンス | 出典表示を条件に利用可能，など |
| データ形式 | CSV |
| 更新頻度 | 年1回 |
| 分析してみたい内容 | 人口密度と地域差の比較 |

**解説**

分析しやすいデータかどうかは，単に面白い内容かどうかでは決まらない．
CSV などの機械可読な形式であること，変数の意味や単位が説明されていること，更新頻度が明確であること，利用条件が確認できることが重要である．
```

````{note} 発展課題2：raw と processed を説明する

次の問いに答えよ．

1. `data/raw` に保存するべきものは何か．
2. `data/processed` に保存するべきものは何か．
3. 元データを直接上書きしない方がよい理由は何か．
4. 加工後データだけを残すと，どのような問題が起こるか．
````

```{dropdown} 解答例

1. `data/raw` には，取得直後のデータを保存する．できるだけ手を加えず，元の状態を保つ．
2. `data/processed` には，前処理，集計，変換などを行った後のデータを保存する．
3. 元データを直接上書きすると，取得時点の状態に戻れなくなり，加工の影響を検証できなくなる．
4. 加工後データだけでは，どのような処理によって作られたのか，元データと比べて何が変わったのかを確認しにくくなる．

**解説**

データ分析では，結果だけでなく，結果に至る過程を説明できることが重要である．
`raw` と `processed` を分けることは，分析過程を追跡可能にするための基本的な工夫である．
```

````{note} 発展課題3：取得操作を関数として考える

本文では，データ取得を
$$
D = G(u, t, \theta)
$$
と表した．

次の問いに答えよ．

1. $u$，$t$，$\theta$ はそれぞれ何を表すか．
2. 実際のデータ取得で，$\theta$ に含まれる条件にはどのようなものがあるか．
3. このように考えることは，再現可能性にどのように役立つか．
````

```{dropdown} 解答例

1. $u$ は取得元，$t$ は取得時点，$\theta$ は取得条件を表す．
2. 地域，期間，対象年齢，ファイル形式，検索条件，API のパラメータなどが考えられる．
3. 取得元，取得時点，取得条件を明確にすれば，同じ条件でデータを再取得しやすくなる．また，結果が変わった場合に，データの更新によるものか，分析プログラムの変更によるものかを切り分けやすくなる．

**解説**

データ取得は，単なるダウンロード作業ではない．
どの条件でデータを得たのかを明示することで，分析全体を検証可能な手続きとして扱えるようになる．
```
