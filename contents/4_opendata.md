## オープンデータとは何か

### 公開されていることとオープンであることの違い
公開データという言葉は広く使われるが，「ウェブ上で見られる」ことと「オープンデータである」ことは同じではない．

一般にオープンデータとは，誰もが利用，複製，再配布，加工等を行えるように，適切なライセンスの下で公開されたデータを指す．ここで重要なのは，単なる閲覧可能性ではなく，**再利用可能性** が明示されていることである．

したがって，ある表がウェブページに掲載されていても，その利用条件が不明なら，自由に分析や再配布を行ってよいとは限らない．

### オープンデータの社会的意義
オープンデータは，行政の透明性向上，科学的検証，市民参加，企業活動，教育など，さまざまな領域に寄与する．

たとえば行政統計が機械可読な形で公開されていれば，地域比較，政策評価，都市計画，市場分析などに利用できる．気象データや感染症データであれば，時系列解析や予測，リスク評価に活用できる．

この意味で，オープンデータは単なる素材ではなく，社会の意思決定や知識生成を支える基盤である．

## データ取得で確認すべき事項

### 出典
データ取得において最初に確認すべきなのは，そのデータの出典である．出典が不明なデータは，正確性や更新性を評価できない．

出典とは，単に URL を指すだけではなく，どの機関・団体・研究プロジェクトが提供しているかという意味を含む．たとえば，行政機関の統計，大学研究室の公開データ，民間企業の提供データでは，品質管理や更新方針が異なる可能性がある．

### ライセンス
次に重要なのがライセンスである．ライセンスは，そのデータをどこまで利用してよいかを定める．

たとえば，次の点が問題となる．

- 商用利用が可能か
- 改変が許可されているか
- 再配布が可能か
- 出典表示が必要か

データ分析の授業では，単に「使えるか」ではなく，「どの条件の下で使えるか」を確認する姿勢が必要である．

### 更新頻度と取得時点
データは固定されたものとは限らない．特に気象，交通，経済，感染症などのデータは継続的に更新される．

このとき重要なのは，分析結果が
$$
R = F(D_t, P)
$$
のように，時点 $t$ のデータ $D_t$ に依存するという事実である．同じプログラム $P$ を使っても，取得時点が異なれば結果 $R$ は変わり得る．

したがって，どの日時に取得したかを記録することは，再現可能性の観点から重要である．

### データ形式
同じ内容のデータでも，CSV，JSON，Excel，HTML 表など，形式が異なれば処理方法は変わる．形式は単なる保存拡張子ではなく，データの構造そのものを反映している．

分析者は，内容だけでなく，構造を理解しなければならない．

## 機械可読性とデータ形式

### 機械可読性とは何か
データが人間に読めることと，機械が処理しやすいことは一致しない．PDF に表が掲載されていても，人には読めても機械的処理は難しいことがある．

一方，CSV や JSON は，プログラムから比較的扱いやすい．この違いは，情報の構造が明示されているかどうかに関係する．

### 構造化データと半構造化データ
代表的には，次のように整理できる．

- CSV：表形式を前提とした構造化データ
- Excel：表形式だが，装飾や複数シートなどを含みうる
- JSON：キーと値の組や入れ子構造を持つ半構造化データ
- HTML：表示目的の構造を持ち，分析用には追加処理が必要な場合が多い
- PDF：人間向け文書形式であり，分析用データとしては扱いにくいことが多い

形式の違いは，後の前処理コストに直結する．したがって，入手可能な形式が複数あるなら，分析しやすい形式を優先するのが合理的である．

## CSVと表データの考え方

### 行と列
CSV は，表形式データを扱う最も基本的な形式の1つである．各行が観測単位，各列が変数を表すことが多い．

たとえば，市区町村別人口データであれば，

- 行：各市区町村
- 列：人口，面積，高齢化率など

のように対応する．

このとき，データ表は数学的には，行集合 $I$ と列集合 $J$ に対する値の対応
$$
X : I \times J \to V
$$
とみなすことができる．ここで $V$ は値の集合である．この見方を持つと，表データは単なる文字の集まりではなく，変数と観測からなる構造体であることが分かる．

### tidy data の考え方
分析しやすい表データには一定の原則がある．代表的には，次の考え方が重要である．

- 1つの列は1つの変数に対応する
- 1つの行は1つの観測に対応する
- 1つの表は1種類の観測単位に対応する

このような整理は，後の可視化や集計を容易にする．授業では今後，前処理の回で詳しく扱うが，第4回の段階でも「取得直後の表はそのまま分析に向いているとは限らない」ことを理解しておく必要がある．

## JSONと階層構造

### JSONの基本構造
JSON は，キーと値の組を基本とする形式であり，リストや入れ子構造を持つことができる．そのため，表形式より柔軟に情報を表現できる．

たとえば，ある地域の気象情報を

- 地域名
- 日時
- 気温
- 降水量
- 観測点情報

のように階層的に表せる．

### 木構造としての理解
JSON の構造は，情報数理的には木構造として理解しやすい．各オブジェクトを節点，キーを枝，値を葉または部分木とみなせば，JSON 全体は rooted tree に近い構造を持つ．

したがって，JSON の処理は「木の中から必要な節点を取り出す操作」として考えることができる．これは表形式データとは異なる視点を必要とする．

### メタデータの重要性

データ本体だけでなく，取得日時，取得元 URL，ライセンス，更新日，変数説明などを残しておくことが望ましい．これらはメタデータである．

分析は，データ本体だけでは十分に解釈できないことが多い．変数名だけを見ても意味が分からない場合や，単位が不明な場合がある．したがって，データ取得時にはメタデータもセットで保存する習慣が重要である．

## Git管理とデータ取得

### 何をGitで管理するべきか

第2回・第3回で学んだ Git は，ここでも重要である．ただし，データ分析では何でも Git に入れればよいわけではない．

通常，Git で管理するものとしては次が考えられる．

* 取得用スクリプト
* 前処理スクリプト
* README や取得手順書
* 小規模なサンプルデータ
* メタデータ

一方，大きな生データや頻繁に更新される取得結果は，そのまま全て Git に入れると管理が難しくなる場合がある．

### 手順を管理するという考え方

ここで重要なのは，「データそのもの」を管理するだけでなく，「どのようにデータを取得したか」という手順を管理することである．

理想的には，取得操作を関数
$$
D = G(u, t, \theta)
$$
とみなせるようにしたい．ここで，

* $u$：取得元
* $t$：取得時点
* $\theta$：取得条件

であり，結果としてデータ $D$ が得られる．

このとき，$u$，$t$，$\theta$ を記録しておけば，後から同様の取得を再実行しやすくなる．

## データ取得の倫理と注意点

### 利用条件の確認

公開されているからといって，自由に何をしてもよいわけではない．特に，個人情報やセンシティブな情報を含むデータには注意が必要である．

授業で扱うのは主に公開データであるが，公開データであっても利用条件や再配布条件は確認する必要がある．

### データ品質の問題

オープンデータは有用である一方，欠損，表記揺れ，更新遅れ，定義変更などの問題を含むことがある．

したがって，分析者は「公開されているから正しい」と無批判に受け入れるのではなく，データ生成過程や定義を確認しなければならない．これは統計的理解や推論の前提に関わる．

## 演習6　整理済みデータを作成する

`population.csv` を基に，人口密度を追加した `data/processed/population_processed.csv` を作成する Python スクリプト `src/process_population.py` を作成せよ．

```python
import csv

input_path = "data/raw/population.csv"
output_path = "data/processed/population_processed.csv"

rows_out = []

with open(input_path, encoding="utf-8") as f:
    reader = csv.DictReader(f)
    for row in reader:
        population = int(row["population"])
        area = float(row["area"])
        density = population / area
        rows_out.append({
            "prefecture": row["prefecture"],
            "city": row["city"],
            "population": population,
            "area": area,
            "density": density
        })

with open(output_path, "w", encoding="utf-8", newline="") as f:
    writer = csv.DictWriter(
        f,
        fieldnames=["prefecture", "city", "population", "area", "density"]
    )
    writer.writeheader()
    writer.writerows(rows_out)

print("processed data saved.")
```

## 発展事項　データは観測の結果である

データ分析では，手元にある表や JSON をそのまま自然物のように扱いがちである．しかし，データは常に何らかの観測・集計・記録・加工の過程を経て生成されている．

したがって，データを理解するには，値だけでなく，その生成過程を考えなければならない．同じ「人口」という変数でも，常住人口なのか推計人口なのかで意味が異なることがある．

この意味で，データ取得は単なるダウンロードではなく，観測体系へのアクセスでもある．

## 発展事項　分析可能性と表現形式

あるデータが存在しても，その表現形式が分析に向いていなければ，計算資源として活用しにくい．

情報数理の観点から見れば，データ形式とは，情報をどのような構造で符号化しているかという問題である．表形式は行列に近い構造を持ち，JSON は木構造に近い．形式の違いは，探索，変換，結合のコストに影響する．

したがって，分析者は，対象データを「何が書いてあるか」だけでなく，「どの構造で表されているか」という観点からも見る必要がある．

* データ取得では，出典，ライセンス，更新頻度，取得時点，形式を確認する必要がある
* CSV は表形式，JSON は階層構造を持つ代表的な形式である
* データ本体だけでなく，取得記録やメタデータを残すことが重要である
* 生データと加工データを分けて保存することで，再現可能性が高まる
* Git では，データそのものだけでなく，取得手順と整理過程を管理することが重要である
