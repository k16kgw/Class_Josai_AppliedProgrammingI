## 最終レポートの目的

各自で問いを設定し，次の一連のデータ分析を行う．

```text
テーマ設定
  ↓
データの探索・取得
  ↓
データの内容と利用条件の確認
  ↓
前処理・結合
  ↓
可視化・比較
  ↓
結果の考察
```

第11回で予告した通り，1種類のデータだけを眺めるのではなく，**2種類以上のオープンデータ**を取得して比較すること．
一見すると関係が明らかでない対象を組み合わせ，どのような関係や共通点，違いがあるかを可視化から検討する．

```{tip} 評価基準
必ずしも相関を発見する必要はない．
比較した結果，「明確な関係は確認できなかった」「期間や地域によって関係が異なった」という結論でもよい．
評価指標は次の9項目（40点）である．

| 観点 | 確認する内容 | 点数 |
| --- | --- | --- |
| (1) 問い　 | 比較する対象と明らかにしたいことが具体的か　　　　　　 | 5点 |
| (2) データ | 出典と利用条件が明確で，問いに適したデータか　　　　　 | 4点 |
| (3) 前処理 | 加工内容とデータを対応付ける方法が説明されているか　　 | 4点 |
| (4) 再現性 | PythonファイルとREADMEから分析を再実行できるか　　　 | 5点 |
| (5) 可視化 | 図の種類，軸，単位，色，凡例が目的に合っているか　　　 | 5点 |
| (6) 結果　 | 図から読み取れる事実を具体的に説明しているか　　　　　 | 4点 |
| (7) 考察　 | 結果と解釈を区別し，別の可能性も検討しているか　　　　 | 5点 |
| (8) 限界　 | データから判断できないことや分析上の制約を示しているか | 4点 |
| (9) 管理　 | フォルダ構成とGit履歴が整理されているか　　　　　　　 | 4点 |
```

## 満たすべき条件

次の条件をすべて満たすこと．

| 条件 | 必須内容 |
| --- | --- |
| (1) 問い | 何を比較し，何を明らかにするかを具体的に示す |
| (2-1) データ | 2種類以上のオープンデータを使用する |
| (2-2) 出典 | データ名，提供者，URL，取得日を記録する |
| (2-3) 利用条件 | ライセンスや利用条件を確認し，レポートに記載する |
| (3-1) 前処理 | 必要な行・列の選択，型変換，欠損処理などを行う |
| (3-2) 対応付け | 時間，地域，カテゴリなど，共通の軸でデータを対応付ける |
| (4) 再現性 | Pythonファイルを実行すると前処理と図を再作成できるようにする |
| (5) 可視化 | 2種類以上の図を作り，少なくとも1つはデータ間の比較に用いる |
| (7) 考察 | 図から直接読み取れる結果を文章で説明する |
| (8) 限界 | データや分析方法だけでは判断できないことを示す |
| (9) Git | 作業の区切りごとにコミットし，変更履歴を残す（commitが2回分あれば良い） |
| (10) 生成AI | 利用した場合は，レポートの最後に利用履歴を記載する |

講義で使用したデータを再利用してもよい．
ただし，講義と同じ図をそのまま提出することは認めない．
自分で設定した問いに合わせた変更をすること．

```{warning} レポートの得点をゼロとする場合
次のいずれかに該当した場合，最終レポートの得点をゼロとする．

- 存在しないサイトを参照している
- 利用条件に抵触したデータの使い方をしている

提出前に記載したURLを実際に開けることと，データの利用方法が公開元の利用条件に沿っていることを必ず確認すること．
```

### データを対応付ける例

| 共通の軸 | 組合せの例 |
| --- | --- |
| 年月 | 物価と完全失業率，人口と住宅価格 |
| 都道府県 | 人口移動と所得，観光客数と宿泊施設数 |
| 市区町村 | 人口と公共施設数，年齢構成と医療施設数 |
| 試合・選手 | 選手位置とイベント，競技成績と選手属性 |

2つのデータで単位や観測間隔が異なる場合は，そのまま結合しないこと．
月次と年次，市区町村と都道府県など，集計単位をそろえてから比較する．

## レポート本文の構成

次の項目を参考にレポートを作成すること．
ただし各項目について厳密に準拠する必要はない．
カッコ〔〕付きの項目は該当する場合のみ考慮すること．

### 体裁

- [ ] 用紙サイズ：A4
- [ ] 文字サイズ：10pt以上かつ12pt以下
- [ ] ページ数：3ページ以上

### テーマと問い

- [ ] 分析の題名
- [ ] 比較する対象
- [ ] 明らかにしたい問い
- [ ] その問いを選んだ理由

### 使用したデータ

データごとに次を記載する．

- [ ] データ名
- [ ] 提供者
- [ ] URL
- [ ] 取得日
- [ ] 〔対象期間・対象地域〕
- [ ] ライセンスまたは利用条件（またはそれを確認できるサイトのURL）
- [ ] 主に使用するデータ（例：降水量・最高気温）

### 分析方法

- [ ] データの取得方法（例：サイトからダウンロード，PythonのスクリプトからAPIを叩いて取得など）
- [ ] 行った前処理
- [ ] 欠損値の扱い
- [ ] 2つ以上のデータをどの列で対応付けたか（例：都道府県が同じ列を参照して対応づけた）
- [ ] 自身で作成した指標や集計値（例：日々のデータから1ヶ月ごとに平均値を計算した列を作成した）
- [ ] 使用した可視化方法と，その方法を選んだ理由

### 結果

- [ ] 作成した図表
- [ ] 図のタイトル，軸ラベル，単位，凡例
- [ ] 図から直接確認できる傾向，共通点，違い

記載された全ての図について，どの部分に注目すべきかを文章で説明すること．

### 考察と限界

- [ ] 問いに対してどのような結果が得られたか
- [ ] 想定と異なった点
- [ ] データの期間，範囲，欠損，測定方法による制約
- [ ] 今回の分析だけでは判断できないこと
- [ ] 次に追加するとよいデータや分析

```{tip} 相関を扱う場合
相関係数だけを示さず，時系列図や散布図も確認すること．
対象期間，共通トレンド，自己相関，擬似相関に注意し，相関関係だけから因果関係を結論づけないこと．
```

### 参考資料

- [ ] 使用したデータ，データ仕様，参考にしたページや文献を一覧にする
- [ ] 提供者・ページ名・URL・参照日を記載すること

### 補足：生成AIの利用履歴

生成AIは自由に利用してよい．
生成AIを利用した場合は，レポートの最後の章を「補足：生成AIの利用履歴」とし，次のいずれかの方法でやり取りの内容が分かる資料を掲載すること．

- 生成AIとのチャットを共有するリンク
- やり取りの内容を確認できるスクリーンショット
- その他，入力した内容と生成AIの回答を確認できる記録

共有リンクを掲載する場合は，教員がリンク先を閲覧できることを提出前に確認すること．
例えばプライベートブラウザでリンクが開けるかを確認すれば良い．
生成AIを利用しなかった場合は，この章に「生成AIは利用していない」と記載すること．

<span style="color:red">生成AIとのやり取りの仕方は評価対象としない．</span>
どのようなやり取りをしていたとしても，その内容によってレポート評価の高低が変わることはない．
提出された利用履歴を確認し，今後の使い方についてのアドバイスをフィードバックするためだけに参照する．

## 提出物の条件

### 問1：分析フォルダのzipファイル

問1へ提出するzipファイルには，次の構成を含めること．

```text
report/
├── .git/             # Gitの変更履歴（隠しフォルダ，`git init`しておけば作成されている）
├── data/
│   ├── raw/          # データ量が大きくなるので，rawフォルダは空にすること
│   └── processed/    # 前処理後・結合後のデータ
├── notebooks/
│   └── analysis.ipynb（観察用，Gitでは管理しない）
├── reports/
│   └── figures/      # レポートで使用した図
├── src/
│   └── vis.py など    # 再実行可能なPythonファイル
├── README.md
└── .gitignore
```

- [ ] データ取得，前処理，可視化の本処理は`src/`内のPythonファイルとして残すこと．
- [ ] Notebookはデータの確認や試行錯誤に使用してよいが，Notebookだけに最終処理を残さないこと．
- [ ] data/rawの生データフォルダは空にすること（データ量が大きくなりWebClassから提出できない可能性がある）
- [ ] README.mdにデータの取得方法を記載すること（教員が手元で生データを取得できるようにすること）

教員がレポート評価の際に各Pythonスクリプトファイルを実行してファイルが作成されることを確認する．
教員の環境が原因で実行できない場合でも正当に評価できるようコードを確認して評価する．
